# Deployment Guide

Target setup: **API on Render or Northflank** (Docker) + **frontend on
Vercel**. Both paths are covered; pick one for the API.

> Important free-tier note: with no `MONGODB_URI`/`QDRANT_URL` configured,
> data lives in process memory - it disappears on every restart/redeploy,
> and Render's free tier sleeps after inactivity. Fine for a demo; add
> MongoDB Atlas + Qdrant Cloud (both have free tiers) for persistence.

---

## 0. Prerequisites

- Push this repository to GitHub (public or private).
- Generate a JWT secret you'll reuse below:
  ```bash
  python -c "import secrets; print(secrets.token_urlsafe(48))"
  ```

## 1a. API on Render (blueprint - easiest)

1. Sign in at [render.com](https://render.com) with GitHub.
2. **New → Blueprint**, select this repository. Render reads
   [render.yaml](render.yaml) and creates the `rag-application-api` service
   (Docker, free plan, health check on `/api/health`).
3. In the service → **Environment**, fill in the blank variables when you
   have them (`MONGODB_URI`, `QDRANT_URL`, `QDRANT_API_KEY`,
   `GEMINI_API_KEY`, `COHERE_API_KEY`). Leave blank to stay on local
   fallbacks. `JWT_SECRET` is auto-generated by the blueprint.
4. Deploy, then verify: `https://<your-api>.onrender.com/api/health`
   should return `{"status":"ok", ...}`.

*(Manual alternative: New → Web Service → your repo, Root Directory
`backend`, Runtime `Docker`, then add the env vars from
`backend/.env.example`.)*

## 1b. API on Northflank (alternative)

1. Sign in at [northflank.com](https://northflank.com), create a project.
2. **Create service → Combined service**:
   - Repository: your GitHub repo, branch `main`
   - Build type: **Dockerfile**, context `/backend`,
     Dockerfile path `/backend/Dockerfile`
   - Port: `8000` (HTTP, publicly exposed)
3. Under **Environment variables**, add `JWT_SECRET`,
   `ENVIRONMENT=production`, `CORS_ORIGINS=https://<your-app>.vercel.app`,
   plus any provider keys you have.
4. Deploy and verify `https://<service>.northflank.app/api/health`.

## 2. Frontend on Vercel

1. Sign in at [vercel.com](https://vercel.com) with GitHub → **Add New →
   Project** → import this repository.
2. Set **Root Directory** to `frontend`. Vercel auto-detects Vite
   (build `npm run build`, output `dist`); `frontend/vercel.json` already
   handles SPA routing.
3. Add environment variable:
   - `VITE_API_URL` = `https://<your-api>.onrender.com` (or the
     Northflank URL) - **no trailing slash**.
4. Deploy. Note your production URL, e.g. `https://rag-studio.vercel.app`.

## 3. Connect the two (CORS)

Back on Render/Northflank, set:

```
CORS_ORIGINS=https://<your-app>.vercel.app
```

(comma-separate to also allow `http://localhost:5173` for local dev
against the deployed API). Redeploy/restart the API.

## 4. Add persistence when ready (both free)

**MongoDB Atlas** - create an M0 free cluster at
[mongodb.com/atlas](https://www.mongodb.com/atlas), create a database user,
allow access from `0.0.0.0/0` (or the host's IPs), copy the connection
string into `MONGODB_URI`. Users, documents metadata, chat history and
refresh tokens become persistent.

**Qdrant Cloud** - create a free 1GB cluster at
[cloud.qdrant.io](https://cloud.qdrant.io), copy the cluster URL into
`QDRANT_URL` and the API key into `QDRANT_API_KEY`. Vectors become
persistent; the collection is created automatically on startup.

**Model keys** - `GEMINI_API_KEY` from
[aistudio.google.com](https://aistudio.google.com/apikey) (free tier),
`COHERE_API_KEY` from [dashboard.cohere.com](https://dashboard.cohere.com)
(trial). Optionally set `EMBEDDING_PROVIDER=gemini` for neural embeddings -
do this **before** indexing documents you care about, since existing
384-d vectors are not reused by the 768-d collection.

## 5. Post-deploy checklist

- [ ] `/api/health` shows the providers you expect
- [ ] Register + login from the Vercel URL works (CORS is correct)
- [ ] Upload → query returns citations
- [ ] `JWT_SECRET` is set (never rely on the ephemeral dev secret)
- [ ] `ENVIRONMENT=production`
- [ ] Real keys exist **only** in the hosting dashboards, never in git
